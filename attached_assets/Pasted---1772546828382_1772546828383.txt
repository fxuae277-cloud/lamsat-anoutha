تمام — واضح من الصور إن “قسم التقارير المالية” شكله ممتاز، لكن منطقيًا وتشغيليًا فيه مشكلتين رئيسيتين تسبب “لا يعمل / لا يظهر تفاصيل”:

التقارير مبنية على تاريخ واحد فقط (Day) بدل من–إلى + فلاتر (فرع/شفت/موظف/طريقة دفع).

صفحات الفئات/المنتجات/الدفع تعرض “ملخّص” بدون Drill-down (تفاصيل العناصر + قائمة عمليات قابلة للفرز والتصدير).

أعطيك الآن إعادة هيكلة كاملة (Information Architecture + UX + منطق تقارير محاسبي مهني) + قائمة تنفيذ واضحة.

1) الهيكلة الجديدة لقسم التقارير (منطق محاسبي احترافي)
A) نظرة عامة (Dashboard)

نطاق تاريخ: من–إلى (افتراضي: اليوم)

فرع: جميع الفروع / فرع محدد

شفت: كل الشفتات / شفت محدد (يتفلتر حسب الفرع والتاريخ)

بطاقات KPI:

إجمالي المبيعات

COGS

إجمالي الربح

المصروفات

صافي الربح

تفصيل طرق الدفع: نقدي/بطاقة/تحويل

زر تصدير: PDF / Excel / CSV

هذه الصفحة لازم تكون “مدخل سريع” وليست تقرير تفصيلي فقط.

B) تقارير المبيعات (Sales Reports)

1) مبيعات يومية/فترة

جدول عمليات المبيعات (كل فاتورة/عملية):

رقم الفاتورة، التاريخ/الوقت، الفرع، الشفت، الموظف، إجمالي، خصم، ضريبة (إن وجدت)، طريقة الدفع، الحالة (paid/cancelled/pending)

تفاصيل الفاتورة (زر “عرض”):

عناصر الفاتورة (المنتج/الكمية/السعر/التكلفة/الربح)

2) تقرير الربحية

Revenue, COGS, Gross Profit, Expenses, Net Profit

رسم بسيط اختياري (اتجاه يومي/أسبوعي)

C) تقارير الدفع (Payments)

هذه لازم تُبنى على “دفعات” وليس فقط “مبيعات”.

فلاتر: من–إلى + الفرع + طريقة الدفع + الموظف + رقم العملية

جدول:

التاريخ/الوقت

المرجع: sale_id / invoice_no

الطريقة: Cash/Card/Bank

المبلغ

رسوم (إن وجدت)

رقم مرجع البنك/البطاقة (bank_txn_id)

مرفق إيصال (اختياري)

ملخص أعلى التقرير:

إجمالي النقدي

إجمالي البطاقة

إجمالي التحويل

الإجمالي الكلي

“تقارير الدفع” هي اللي تثبت التدفق النقدي فعلًا — لازم تكون قابلة للتدقيق.

D) تقارير الشفتات (Shifts)

فلاتر: من–إلى + الفرع + الجهاز/التيرمنال + الموظف

جدول الشفت:

فتح الشفت: وقت + رصيد افتتاحي

إغلاق الشفت: وقت + النقد الفعلي + النقد المتوقع + فرق الصندوق

إجمالي مبيعات الشفت حسب طرق الدفع

المصروفات داخل الشفت

Drill-down: “تفاصيل الشفت”

كل الفواتير داخل الشفت

كل المصروفات داخل الشفت

هذا التقرير يحل 80% من مشاكل المحاسبة اليومية.

E) تقارير الفئات (Categories)

بدل “اختر التاريخ” وتطلع صفحة فاضية — لازم:

فلاتر: من–إلى + الفرع

جدول:

الفئة

إجمالي المبيعات

الكمية المباعة

COGS

إجمالي الربح

الهامش %

Drill-down: عند الضغط على الفئة

أفضل المنتجات داخل الفئة

تفاصيل المبيعات (اختياري)

F) تقارير المنتجات (Products)

فلاتر: من–إلى + الفرع + (بحث/باركود/فئة)

جدول:

المنتج

الكمية المباعة

المبيعات

COGS

الربح

الهامش %

Drill-down للمنتج:

قائمة الفواتير التي ظهر فيها المنتج + الكميات والأسعار

G) مقارنة الفروع (Branch Comparison)

فلاتر: من–إلى (إلزامي)

جدول مقارنة:

الفرع | المبيعات | COGS | الربح | المصروفات | صافي الربح | الهامش %

إضافة مهمة:

“عرض تفاصيل الفرع” → يحول لتقرير مبيعات الفرع بنفس المدة.

2) قواعد UX لازم تثبت في كل التقارير (عشان ما “تكون فاضية”)

كل تقرير لازم فيه:

تاريخ من–إلى (إجباري)

فرع (اختياري)

زر “تطبيق” Apply (لا تعتمد على تغيير التاريخ فقط)

حالة فارغة ذكية

بدل “اختر التاريخ” فقط:

“لا توجد عمليات في هذه الفترة — جرّب تغيير المدة أو الفرع”

دقة الوقت

استخدم from=00:00 to=23:59 لليوم

التصدير

التصدير يصدّر “نفس الفلاتر المطبقة” (مو مهم يتصدّر كل شيء)

3) لماذا تقاريرك الآن لا تعمل غالبًا؟ (أسباب واقعية)

تعتمد على تاريخ واحد وبعض الصفحات لا ترسل الطلب للـ API إلا عند وجود تاريخ/اختيار إضافي.

التقارير مثل “الفئات/المنتجات” تحتاج JOIN مع sales_items + products + categories + costs — لو الـ Query ناقص يطلع 0 أو فاضي.

“الدفع” إذا لم يكن عندك جدول payments مستقل (أو لم تحفظ payment_method لكل عملية بشكل صحيح) سيظهر كل شيء 0.

4) خطة تنفيذ سريعة (Checklist)

المرحلة 1 (أساس التقارير)

توحيد الفلاتر: date_from, date_to, branch_id, shift_id, category_id, product_id, payment_method

توحيد Response:

summary + rows + totals

المرحلة 2 (تفاصيل Drill-down)

endpoints للتفاصيل:

invoice details

shift details

category breakdown

product breakdown

المرحلة 3 (التصدير)

CSV/Excel/PDF لنفس query parameters
